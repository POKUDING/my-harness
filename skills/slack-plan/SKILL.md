# Slack List → 작업 계획서 생성

Slack List의 수정요청/작업 아이템을 가져와서 구조화된 작업 계획서를 생성한다.

## 사용자 입력 UI (v0.17+, v0.19+)

URL 입력 같은 자유 텍스트는 일반 대화로 받되, **분기 선택**과 **명확화 질문**은 `AskUserQuestion`을 사용한다.

| 사용 시점 | 용도 | 예 |
|----------|------|----|
| Step 2.5.2 진입 분기 | brainstorming 진행 / 스킵 / 직접 추리기 | 후보 N개 안내 |
| Step 2.5.3 라운드 질문 | 답이 2~4개 보기로 떨어지는 명확화 질문 | 우선순위 재확인, 범위 결정, 완료 기준 |
| Step 2.5.3 항목 종료 분기 | 다음 후보 / 멈춤 / 한 라운드 더 | 매 항목 명확화 종료 시 |
| Step 2.5.3 라운드 4 진입 시 | 충분히 명확? / 보류? | 상한 도달 시 자동 분기 |
| Step 4 다음 단계 분기 | /plan-execute 실행 / 검토 / 종료 | 계획서 생성 후 |

자유 텍스트 질문은 답이 정형화되지 않을 때만 사용 (구체적 동작 묘사·링크 공유 등).

## 사용법

```
/slack-plan <SLACK_LIST_URL 또는 LIST_ID>
```

## 실행 흐름

### Step 0: 설정 확인 및 URL 결정

`.harness/config.env` 파일이 존재하고 `SLACK_BOT_TOKEN` 또는 `SLACK_USER_TOKEN` 키가 포함되어 있는지 확인한다.

설정이 없으면:
```
Slack 토큰이 설정되지 않았습니다.
먼저 /slack-setup 을 실행해 토큰을 저장하세요.
```
라고 안내하고 실행을 중단한다.

**사용할 URL 결정 (우선순위 순):**
1. 인자로 URL이 전달된 경우 → 해당 URL 사용 + `.harness/config.env`의 `SLACK_LIST_URL`을 업데이트
2. 인자 없음 + `config.env`에 `SLACK_LIST_URL` 존재 → 저장된 URL 사용
3. 인자 없음 + `SLACK_LIST_URL` 없음 → 사용자에게 URL 입력 요청 후 `config.env`에 저장

URL을 새로 저장/업데이트할 때:
```bash
# SLACK_LIST_URL이 없으면 추가, 있으면 교체
grep -q "^SLACK_LIST_URL=" .harness/config.env \
  && sed -i '' "s|^SLACK_LIST_URL=.*|SLACK_LIST_URL=<URL>|" .harness/config.env \
  || echo "SLACK_LIST_URL=<URL>" >> .harness/config.env
```

### Step 1: Slack List 데이터 가져오기

아래 명령으로 fetch 스크립트를 실행하여 Slack List 데이터를 가져온다.

```bash
python3 skills/slack-plan/scripts/fetch_slack_list.py "<결정된 URL>"
```

- stdout으로 출력된 JSON을 읽는다.
- 에러 발생 시 stderr 메시지를 사용자에게 전달하고 중단한다.

### Step 2: 데이터 분석 및 계획서 작성

가져온 JSON 데이터의 `items` 배열을 분석하여 아래 형식의 markdown 계획서를 작성한다.

**출력 파일**: `docs/plans/slack-list-<list_id>-plan.md`

(list_id는 JSON의 `metadata.list_id`에서 가져온다)

### 계획서 필수 섹션

```markdown
# 작업 계획서: <리스트 제목 또는 list_id>

생성일: YYYY-MM-DD
원본: <Slack List URL>
총 아이템: N개

---

## 1. 개요 및 범위

이 계획서의 목적과 다루는 범위를 1-2문단으로 서술한다.

## 2. 원문 요청 요약

Slack List의 각 아이템을 원문 그대로 요약한다.
원문에 없는 내용을 추가하지 않는다.

`record_id`는 JSON 응답의 각 item에 있는 `record_id` 필드를 그대로 기록한다. `/slack-review`가 아이템을 식별하는 데 사용한다.

| # | record_id | 원문 요청 | 담당자 | 상태 |
|---|-----------|----------|--------|------|
| 1 | Rec... | ... | ... | ... |

## 3. 실행 TODO

비슷한 요청은 묶어서 정리한다. 각 TODO는 아래 형식을 따른다.

### TODO-001: <제목>

- **설명**: 구체적인 작업 내용
- **우선순위**: P0 / P1 / P2
- **예상 작업 범위**: 영향받는 파일/모듈/시스템
- **선행 조건**: 이 작업 전에 완료되어야 할 것
- **완료 기준**: 이것이 충족되면 완료로 간주
- **관련 원문**: #N (원문 요청 요약 테이블의 번호)
- **Slack record_id**: Rec... (관련 Slack 아이템의 record_id, 여러 개면 쉼표 구분)

## 4. 우선순위 분류

### P0 (긴급 - 즉시 처리)
- TODO-XXX: ...

### P1 (높음 - 이번 스프린트)
- TODO-XXX: ...

### P2 (보통 - 다음 스프린트 이후)
- TODO-XXX: ...

## 5. 단계별 작업 계획

시간순으로 어떤 순서로 작업할지 단계별로 서술한다.

### Phase 1: <제목>
- 작업 내용
- 예상 산출물

### Phase 2: <제목>
- ...

## 6. 확인 필요 사항

원문이 모호하거나 추가 확인이 필요한 항목을 나열한다.
요청 원문에 없는 사실은 단정하지 않는다.

- [ ] <확인 필요한 내용> (관련: TODO-XXX)
- [ ] ...

## 7. 리스크 / 의존성

### 리스크
- <리스크 항목>: 영향도, 대응 방안

### 의존성
- <의존하는 시스템/팀/작업>

## 8. 완료 기준 / 테스트 포인트

전체 작업이 완료되었다고 판단할 수 있는 기준.

- [ ] <완료 기준 1>
- [ ] <완료 기준 2>
- [ ] ...
```

### Step 2.5: Targeted Brainstorming (NEW, v0.19+)

Step 2의 계획서 초안을 받아, 모호한 TODO를 자동 식별하고 짧은 대화로 명확화한다. 명확화 결과는 계획서의 해당 TODO 블록 아래에 append한다.

#### Step 2.5.1: 자동 휴리스틱으로 brainstorming 후보 선정

다음 룰 중 **하나라도** 매치하면 brainstorming 후보:

| ID  | 룰 |
|-----|-----|
| T1  | 우선순위 P0 |
| T2  | 계획서 "확인 필요 사항"으로 자동 분류된 항목 |
| T3  | 원문 길이 ≤ 30자 + 모호 동사 매치 |
| T4  | 영향 범위 큰 키워드 포함 |
| T5  | 담당자 미지정 + P1 이상 |

**T3 모호 동사 화이트리스트:** `개선`, `고쳐`, `검토`, `정리`, `확인`, `봐줘`, `처리`, `해결`, `손봐`

**T4 큰 영향 키워드 화이트리스트:** `마이그레이션`, `migration`, `권한`, `permission`, `auth`, `결제`, `payment`, `배포`, `deploy`, `스키마`, `schema`, `인덱스`, `index`

**상한:** 후보 ≤ 5개. 5개 초과 시 우선순위 `T2 > T1 > T4 > T3 > T5` 순으로 잘라서 상위 5개. 나머지는 계획서의 "확인 필요 사항"에 그대로 남긴다.

후보 0건이면 Step 2.5 전체를 스킵하고 `"휴리스틱 매치 0건 → brainstorming 생략"`이라고 한 줄 알린 뒤 바로 Step 3으로 진행.

#### Step 2.5.2: 진입 분기 (AskUserQuestion)

후보가 1개 이상이면 사용자에게 진행 의사를 묻는다.

```python
AskUserQuestion({
  questions: [{
    question: "자동 휴리스틱이 N개 TODO를 brainstorming 후보로 선정했습니다. 진행 방식은?",
    header: "Brainstorming",
    options: [
      { label: "진행 (Recommended)",
        description: "후보를 하나씩 깊이 우선으로 명확화. 각 항목 끝에 stop 가능." },
      { label: "스킵하고 계획서 그대로 마무리",
        description: "현재 계획서를 그대로 사용. 모호 항목은 '확인 필요 사항'에 남음." },
      { label: "직접 추리기",
        description: "후보 목록을 보여주고 multi-select로 일부만 골라 진행." }
    ],
    multiSelect: false
  }]
})
```

- 1번 선택 시: Step 2.5.3로 진행 (전체 후보).
- 2번 선택 시: Step 2.5 전체 종료, Step 3으로 진행. 후보들은 그대로 "확인 필요 사항"에 남는다.
- 3번 선택 시: 후보 목록을 multi-select 형태(`AskUserQuestion`의 `multiSelect: true`)로 다시 묻고, 선택된 항목만 Step 2.5.3 대상.

#### Step 2.5.3: 항목별 깊이 우선 라운드

후보 TODO 하나씩 다음 사이클을 반복:

##### A. 라운드 단위 질문

각 라운드는 한 번에 한 질문. 도구 선택:

- **AskUserQuestion** — 답이 2~4개 보기로 깔끔하게 떨어지면 사용. 각 옵션의 `description` 필드에 트레이드오프 명시.
  - 예: 우선순위 재확인(P0/P1/P2), 범위 결정(페이지 한정/관련 페이지 다수/백엔드까지), 완료 기준(시각 컨펌/자동 테스트/단순 배포).
- **자유 텍스트 질문** — 답이 정형화 안 되는 경우.
  - 예: "구체적으로 어떤 동작이 잘못되고 있나요?", "원하는 결과 화면을 한 줄로 묘사해주세요", "관련된 PR/이슈/스레드 링크".

도구 선택은 Claude의 자체 판단. 답을 받으면 다음 라운드 또는 종료 판단으로 진행.

##### B. 종료 판단 (Claude 자체 판단)

다음 3요소가 모두 채워지면 명확화 종료:

- **행위**: 무엇을 할지 (동사 + 대상)
- **범위**: 어디까지 영향이 가는지 (파일/모듈/서비스 단위)
- **완료 기준**: 어떻게 확인하는지 (시각 컨펌 / 자동 테스트 / 배포 / 메트릭)

누락된 칸이 있으면 다음 라운드로 진행. 한 항목당 최대 3 라운드 권장.

##### C. 라운드 4 진입 시 자동 분기

3 라운드를 다 썼는데 여전히 누락된 칸이 있으면 다음을 호출:

```python
AskUserQuestion({
  questions: [{
    question: "TODO-XXX 명확화가 3 라운드 만에 완결되지 않았습니다. 어떻게 처리할까요?",
    header: "Round limit",
    options: [
      { label: "충분히 명확함 — 종료",
        description: "현재까지의 답변으로 만족. 통합 형식으로 저장 후 다음 후보로." },
      { label: "보류",
        description: "부분 명확화 결과만 저장하고, 이 항목은 '확인 필요 사항'으로 강등. 다음 후보로." },
      { label: "한 라운드 더",
        description: "1번만 더 질문하고 그 답으로 종료." }
    ],
    multiSelect: false
  }]
})
```

##### D. 명확화 결과 통합 형식

명확화가 종료되면 계획서의 해당 TODO 블록 아래에 다음 형식으로 append. **기존 TODO 필드는 절대 자동 수정하지 않는다.**

```markdown
### TODO-001: 사용자 프로필 개선

- **설명**: ...                  ← 기존 필드 (변경 없음)
- **우선순위**: P1
- **예상 작업 범위**: ...
- **선행 조건**: ...
- **완료 기준**: ...
- **관련 원문**: #3
- **Slack record_id**: Rec0K123ABC

#### 명확화 결과 (Brainstorming, YYYY-MM-DD)

**Q1.** 이 "개선"의 구체적 행위가 무엇인지요? (필드 추가 / 레이아웃 변경 / 데이터 표시 항목 변경)
**A1.** 데이터 표시 항목 변경. 닉네임·가입일 노출, 이메일은 숨김.

**Q2.** 영향 범위는 프로필 페이지 한정인가요, 아니면 카드 컴포넌트가 노출되는 모든 곳인가요?
**A2.** 카드 컴포넌트 전체. 댓글 hover, 멘션 미리보기 포함.

**Q3.** (자유 텍스트) 완료 확인 방식?
**A3.** 디자인팀 시각 컨펌 + 댓글 hover에서 이메일 안 보이는지 확인.

---

**명확화 요약**:
- 행위: 카드 컴포넌트의 표시 필드 개편 (닉네임·가입일 추가, 이메일 제거)
- 범위: 프로필 페이지 + 댓글 hover + 멘션 미리보기
- 완료 기준: 디자인 컨펌 + 이메일 비노출 시각 확인

**기존 필드 갱신 권장**:
- "예상 작업 범위" → 카드 컴포넌트 출현 전 위치
- "완료 기준" → 디자인 컨펌 + 이메일 비노출
```

규칙:

- Q&A는 그대로 보존 (시간 지나서 의도 추적 가능).
- "명확화 요약"은 plan-execute가 빠르게 읽을 수 있게 별도 블록으로 항상 작성.
- "기존 필드 갱신 권장"은 텍스트 가이드 — 자동 갱신 안 함.
- 계획서의 "확인 필요 사항" 항목 중 brainstorming으로 해소된 항목은 자동 제거.
- **매 라운드 종료 시점마다** 파일을 즉시 저장(Edit/Write)하여 도중 세션 중단에도 결과 유지.

##### E. 항목 종료 분기 (Stop)

매 항목 명확화 종료 시점에:

```python
AskUserQuestion({
  questions: [{
    question: "TODO-XXX 명확화 완료. 다음은?",
    header: "Next",
    options: [
      { label: "다음 후보로 진행 (Recommended)",
        description: "남은 brainstorming 후보 중 첫 번째로 이동." },
      { label: "여기서 멈추고 계획서 마무리",
        description: "남은 후보들은 '확인 필요 사항'에 push. Step 3으로 즉시 진행." },
      { label: "이 항목 한 라운드 더",
        description: "추가 질문이 필요. 같은 항목을 한 번 더 라운드 진행." }
    ],
    multiSelect: false
  }]
})
```

- 1번 선택: 다음 후보가 있으면 그것으로 이동. 없으면 Step 3으로 자동 진행.
- 2번 선택: 처리되지 않은 나머지 후보를 계획서의 "확인 필요 사항" 섹션에 push (이미 거기 있던 항목은 중복 추가 안 함). Step 3으로 진행.
- 3번 선택: 같은 항목으로 라운드 1회 더. 4 라운드 도달 시 라운드 4 분기(C)와 동일 처리.

##### F. 재실행 시 자동 제외

`/slack-plan`을 같은 List에 다시 실행할 때, 계획서에 이미 `#### 명확화 결과 (Brainstorming, ...)` 섹션이 있는 TODO는 Step 2.5.1의 후보 선정에서 자동 제외한다.

판정 룰: TODO 블록 안에 `#### 명확화 결과 (Brainstorming` 헤더가 존재하면 "이미 명확화됨"으로 간주.

### Step 3: 상태를 진행중으로 변경

계획서에 포함된 아이템들의 상태를 `진행중`으로 변경한다.

변경 대상: Step 1에서 가져온 아이템 중 상태가 완료가 아닌 항목
- 완료로 간주하는 값: `완료`, `done`, `complete`, `completed`, `백엔드 배포완료`, `이슈아님`
- 그 외 상태는 모두 `진행중`으로 변경

```bash
python3 skills/slack-plan/scripts/update_slack_list.py <LIST_ID> \
  --record Rec... Rec... \
  --status "진행중"
```

상태 변경에 실패한 아이템이 있으면 경고를 표시하되 나머지는 계속 진행한다.

완료 후 사용자에게 알린다:
```
상태 변경: N개 아이템 → 진행중
  - <아이템 요약 1>
  - <아이템 요약 2>
  ...
```

### Step 4: 다음 단계 분기 (AskUserQuestion)

계획서 생성과 상태 변경이 끝나면 자연스러운 다음 액션을 물어본다:

```
AskUserQuestion({
  questions: [{
    question: "계획서가 생성됐습니다. 다음 단계는?",
    header: "Next step",
    options: [
      { label: "지금 /plan-execute 실행 (Recommended)",
        description: "생성된 TODO들을 의존성 분석 후 병렬 실행. 구현이 자동으로 시작됨." },
      { label: "계획서 먼저 검토",
        description: "생성된 docs/plans/... 파일을 먼저 열어 확인. 수동으로 수정한 뒤 /plan-execute 권장" },
      { label: "여기서 마무리",
        description: "이번엔 계획서 생성까지만. 나중에 필요하면 /plan-execute 실행" }
    ],
    multiSelect: false
  }]
})
```

"/plan-execute" 선택 시 Claude가 바로 해당 스킬 호출.
"계획서 검토" 선택 시 파일 경로만 안내하고 세션 종료.

## 분석 규칙

1. **비슷한 요청은 묶는다**: 같은 페이지, 같은 기능, 같은 모듈에 대한 요청은 하나의 TODO로 합친다.
2. **원문 존중**: 요청 원문에 없는 사실은 단정하지 않는다. 추측이 필요하면 "확인 필요 사항"에 넣는다.
3. **우선순위 기준**:
   - P0: 서비스 장애, 데이터 손실, 보안 이슈
   - P1: 사용자에게 직접 영향을 주는 기능 버그/개선
   - P2: 내부 개선, 리팩토링, 문서화
4. **모호한 항목**: "확인 필요"로 분리하고 어떤 점이 모호한지 명시한다.

## 사전 요구사항

Slack 토큰 및 List URL은 `/slack-setup`으로 설정한다. 스크립트는 `.harness/config.env`에서 `SLACK_BOT_TOKEN`/`SLACK_USER_TOKEN`을 자동으로 로드한다.

Slack App에 필요한 스코프: `lists:read`, `lists:write`.
