# Maintainability Rules — 유지보수성 판정 규칙

## 반드시 지적 (Must Report)

### 1. 책임 과다 혼합
하나의 함수/클래스에 관련 없는 여러 책임이 섞여 있다.

**탐지 패턴:**
- 함수 내에서 HTTP 요청 파싱 + 비즈니스 규칙 + DB 저장이 순차적으로 수행
- 하나의 클래스가 데이터 검증 + 변환 + 외부 API 호출 + 캐싱을 모두 담당
- 콜백/핸들러 안에 비즈니스 로직이 직접 작성됨

**severity:** Major (구조적 문제), Minor (경미한 혼합)

### 2. 의미상 로직 중복
단순 문자열 중복이 아닌, 동일한 비즈니스 의도를 가진 코드가 여러 곳에 존재한다.

**탐지 패턴:**
- validation 규칙 중복: 동일한 필드 검증 로직이 컨트롤러, 서비스, 미들웨어에 분산
- 에러 처리 패턴 중복: try-catch + 로깅 + 응답 변환이 동일한 패턴으로 반복
- 상태 변경 로직 중복: 동일한 상태 전이를 여러 엔드포인트에서 독립 구현
- API 응답 매핑 중복: 같은 엔티티를 DTO로 변환하는 로직이 여러 곳에

**severity:** Major (핵심 로직 중복), Minor (보조 로직 중복)

### 3. 산탄총 수술 구조
요구사항 하나를 수정할 때 여러 파일/모듈을 동시에 수정해야 한다.

**탐지 패턴:**
- 새로운 필드를 추가할 때 5곳 이상 수정 필요
- 상수/설정 값이 여러 파일에 하드코딩
- 변경 포인트가 코드베이스 전반에 흩어져 있음

**severity:** Major

### 4. 테스트 불가 구조
외부 의존성이 하드코딩되어 단위 테스트가 불가능하다.

**탐지 패턴:**
- 함수 내에서 직접 new로 외부 서비스 인스턴스 생성
- 전역 싱글톤에 직접 접근
- 파일 시스템/네트워크/DB에 대한 하드코딩된 호출이 비즈니스 로직 안에

**severity:** Major (핵심 로직), Minor (보조 로직)

### 5. 과도한 조건 분기
3단계 이상 중첩된 조건문, 또는 가독성을 심각하게 해치는 분기 구조.

**탐지 패턴:**
- if > if > if > 실제 로직
- 하나의 함수에 10개 이상의 if/else 브랜치
- switch문의 case마다 50줄 이상의 로직

**severity:** Major (5단계+), Minor (3~4단계)

## 신중히 제안 (Suggest Carefully)

다음 항목은 맥락을 충분히 고려한 후에만 제안한다:

### 취향 수준의 분리
- 현재 잘 동작하는 코드를 "더 깔끔해질 수 있다"는 이유만으로 분리 제안하지 않음
- 실제 변경 빈도와 변경 비용을 근거로 제안

### 과잉 추상화
- 현재 1곳에서만 사용되는 코드를 인터페이스로 분리하라고 하지 않음
- "나중에 필요할 수 있다"는 근거로 추상화 요구하지 않음
- 프로젝트 규모가 작으면 직접적 구현을 우선

### 과도한 쪼개기
- 3줄짜리 로직을 별도 함수로 추출하라고 하지 않음
- 분리하면 오히려 호출 체인이 복잡해지는 경우 지적하지 않음

### 실익 없는 공통화
- 2곳 미만의 중복은 "아직 공통화할 시점이 아님"으로 판단
- 로직이 곧 분기될 가능성이 있는 경우 공통화 제안하지 않음

## 판단 프레임워크

finding을 작성할 때 다음을 자문한다:

1. **이 문제가 실제로 변경 비용을 높이는가?** → 아니면 무시
2. **이 PR이 문제를 도입했는가, 악화시켰는가?** → 아니면 followup
3. **제안하는 수정이 현재 프로젝트 규모에 적절한가?** → 아니면 수준 조절
4. **수정했을 때 회귀 위험은 없는가?** → 있으면 scope 조절
