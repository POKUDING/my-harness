# Severity Guide — 심각도 판정 기준

## 심각도 레벨

### Critical
즉시 수정하지 않으면 장애, 보안 침해, 데이터 손상이 발생한다.

- 서비스 중단 직결 (무한 루프, OOM, 크래시)
- 보안 취약점 즉시 악용 가능 (인증 우회, SQL Injection, RCE)
- 데이터 손실/손상 (잘못된 삭제, 트랜잭션 누락)
- 심각한 기능 오류 (결제 금액 오류, 권한 우회)

### Major
방치하면 팀의 생산성을 저하시키거나, 특정 조건에서 사용자에게 영향을 준다.

- 유지보수성 구조적 저하 (SRP 위반, 핵심 로직 중복, 강한 결합)
- 설계 문제 (변경 시 산탄총 수술 필요)
- 성능/안정성상 의미 있는 리스크 (N+1 쿼리, 에러 처리 누락)
- 조건부 오동작 (특정 입력에서 버그)

### Minor
개선하면 코드 품질이 향상되지만 즉각적 위험은 낮다.

- 경미한 네이밍 개선
- 선택적 함수 추출
- 작은 중복 (2곳 미만)
- 극히 드문 edge case
- 에러 메시지 개선

### Nit
취향/스타일 수준. 팀 컨벤션에 따라 무시 가능.

- 변수명 선호도
- 코드 정렬/공백
- import 순서
- 주석 유무/표현

## 에이전트별 severity 경향

| 에이전트 | 주로 Critical | 주로 Major | 주로 Minor |
|---------|-------------|-----------|-----------|
| Correctness | 장애 직결 로직 오류 | 조건부 오동작 | 드문 edge case |
| Reliability | 서비스 중단, 데이터 손실 | 에러 처리 누락, 리소스 누수 | 로깅 부족 |
| Security | 인증 우회, Injection | XSS, 민감정보 노출 | 보안 모범사례 |
| Performance | OOM, 타임아웃 | N+1, 유의미한 지연 | 마이크로 최적화 |
| Maintainability | (거의 없음) | SRP 위반, 핵심 중복 | 선택적 개선 |

## scope 판정 (fix_now vs followup)

| scope | 기준 |
|-------|------|
| **fix_now** | 이 PR에서 수정 가능 + 수정하지 않으면 문제 악화 |
| **followup** | 기존 구조 문제 + 이 PR에서 해결하기엔 범위 과도 |

### 판정 원칙
- Critical은 거의 항상 `fix_now`
- Major 중 이 PR의 변경이 도입한 문제 → `fix_now`
- Major 중 기존에 존재하던 문제 → `followup` (단, 악화시키면 `fix_now`)
- Minor/Nit은 거의 항상 `followup` 또는 무시
